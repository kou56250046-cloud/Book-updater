## 元の理論（Martinの主要原則）

Robert C. Martinが2008年に提唱したクリーンコードの主要な考え方。

- **意味のある名前** — 変数・関数名は意図を明確に表現する
- **小さな関数** — 関数は1つのことだけをすべき
- **コメントは最小限** — コード自体が語るべき
- **エラーハンドリング** — 例外を適切に使い、エラーを無視しない
- **単一責任の原則（SRP）** — クラスは1つの変更理由だけを持つ

## 最新解釈：AI補助コーディング時代の文脈で

### 「小さな関数」の再解釈

OusterhoutはClean Codeの「関数を極限まで小さく」という原則に異議を唱える。**深いモジュール**（Deep Module）の概念により、関数が小さすぎると呼び出し関係が複雑化し、認知負荷が上昇することを示した。

> AIコーディング（GitHub Copilot、Claude等）の登場で、「小さな関数＝再利用しやすい」という前提が崩れた。AIは文脈を理解して長い関数でも補完できる一方、細切れ関数は生成AIのコンテキスト管理を困難にする。

### 「コメントは最小限」の再考

AI補助コーディング環境では、**JSDocやPythonのdocstring形式のコメントがAIの精度を向上させる**ことが実証されている。型情報・期待する動作・エッジケースをコメントに書くことで、Copilotの提案品質が大幅に改善する。

「コードが語るべき」という原則は有効だが、**AIとのコラボレーションを前提とすると、意図を明示するコメントは投資対効果が高い**。

### エラーハンドリングの進化

Pragmatic Programmerの最新版が強調する「クラッシュアーリー」の原則と、分散システム時代の**可観測性（Observability）**の組み合わせ。エラーはキャッチして握りつぶすのではなく、構造化ログ・トレーシングとセットで設計する。

## 参照書籍からの補足

- **A Philosophy of Software Design** — 複雑さとの戦いを中心に据え、「深いモジュール」vs「浅いモジュール」の概念でClean Codeを補完する視点を提供。
- **Pragmatic Programmer（20th）** — DRY原則・直交性・テキストとしてのすべてなど、時代を超えた原則の現代的適用例を追加。

## まとめ

Clean Codeの原則は今も価値があるが、AI補助コーディング・分散システム・型システムの進化により、**「小さく分割する」より「深く設計する」**、**「コメントを書かない」より「意図を明示する」**という方向に最適解がシフトしている。
